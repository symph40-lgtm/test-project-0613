# Roadmap

> Vibe Build · roadmap 산출물
> 작성일: 2026-06-14
> 기반: docs/PRD.md (v2), docs/userflow.md, docs/idea.md

## Tech Stack

코드베이스와 설정 파일에서 확인된 스택만 기록한다.

- **언어:** TypeScript
- **프레임워크:** Next.js 16 (App Router, Turbopack)
- **UI:** React 19, Tailwind CSS v4 (`@tailwindcss/postcss`)
- **아이콘:** lucide-react
- **폰트:** Pretendard (docs/DESIGN.md 지정 — 아직 코드에 적용 전)
- **백엔드/DB:** Supabase 패키지(`@supabase/supabase-js`, `@supabase/ssr`) 설치됨. 단, 클라이언트 초기화·Auth·DB 스키마·RLS는 아직 미구현. `.env.example`만 존재하고 실제 프로젝트 연결은 미정.
- **품질 장치:** ESLint(`eslint-config-next`) 설정됨. 테스트 러너·CI는 없음.

## Current Implementation Snapshot

현재 레포는 `vibe-build:scaffold`로 생성한 **기본 Next.js 앱 상태**다.

- `app/layout.tsx`, `app/page.tsx`(create-next-app 기본 랜딩)만 존재한다.
- 서비스 고유 라우트·페이지·컴포넌트·상태 관리가 전혀 없다.
- Supabase 클라이언트 초기화 코드, 인증 흐름, 데이터 모델(테이블·마이그레이션)이 없다.
- API route / server action / 외부 데이터 연동이 없다.
- 디자인 시스템(docs/DESIGN.md의 토큰·컴포넌트)이 코드에 반영되어 있지 않다.

결론: **PRD의 모든 Functional Requirement(FR-001~FR-036)가 미충족**이며, 구현은 사실상 0에서 시작한다.

## Requirement Gap Analysis

- **충족됨:** 없음
- **부분 충족:** 없음
- **미충족:** FR-001 ~ FR-036 (전체 36개)
- **확인 필요:** 없음 (구현 자체가 시작되지 않아 상태가 명확)

다만 다수 FR이 **외부 시장 데이터(실시간 시세·매크로·뉴스·실적 캘린더)** 와 **AI 종합 판단 로직**에 의존한다. 데이터 소스·판정 모델은 PRD 오픈 이슈로 미정이므로, 해당 FR을 다루는 마일스톤은 "데이터 공급 가정"을 명시한 채 UI·입력·상태·표현 계층을 우선 완결하는 방식으로 진행한다(아래 Notes 참조).

## Milestone 1 - 이용 신청·관리자 승인 게이트 (Access Gate & Foundation)

### Goal
이용 권한이 없는 방문자가 신청하고, 관리자가 검토·승인하면 승인된 사용자만 핵심 기능에 접근하는 게이트를 end-to-end로 완성한다. 동시에 이후 모든 화면이 공유할 디자인 시스템·앱 셸·Supabase 인증/권한 기반을 마련한다.

### User Value
- 방문자: 이용을 신청하고 대기/승인/거절 상태와 사유를 확인할 수 있다.
- 관리자: 신청을 한곳에서 검토하고 승인·거절하며 결과를 통지할 수 있다.
- 승인된 사용자만 서비스에 진입하므로, 검증된 사용자 기반 위에서 다음 기능을 쌓을 수 있다.

### Scope
- 디자인 시스템 토큰화(docs/DESIGN.md의 색·타이포·간격), Pretendard·lucide-react 적용, 공통 레이아웃/앱 셸, 빈/로딩/오류 상태 패턴 정립.
- Supabase 클라이언트(서버·클라이언트) 초기화, 사용자 인증, 관리자/일반 역할 분리.
- 데이터 모델: 사용자, 이용 신청(상태: 대기/승인/거절·사유), 관리자 역할.
- 화면: 이용 신청 화면, 승인 대기/상태 화면, 관리자 신청 관리 화면.
- 접근 제어: 미승인 사용자가 핵심 기능 라우트에 직접 접근 시 신청/상태 화면으로 안전하게 차단.
- 승인·거절 결과의 앱 내 통지(외부 채널 통지는 M3의 알림 채널과 연계, 본 마일스톤은 앱 내 상태로 폴백).

### Out of Scope
- 종목 입력·브리핑 등 핵심 기능 자체(M2~).
- 외부 SMS/이메일 발송 인프라(통지는 앱 내 상태 우선, 외부 채널은 M3).
- 거절 후 재신청 정책·권한 만료 자동화의 정교한 운영 규칙(기본 동작만, 정책은 오픈 이슈).

### Functional Requirements Covered
FR-030, FR-031, FR-032, FR-033, FR-034, FR-035, FR-036

### Completion Criteria
- 방문자가 이용 신청 화면에서 이름·연락처·동의를 제출하면 신청이 "대기 중"으로 저장되고 상태 화면에서 확인된다.
- 미승인(대기/거절) 사용자가 핵심 기능 URL에 직접 접근하면 신청/상태 화면으로 리다이렉트된다.
- 관리자 인증을 통과한 사용자만 신청 관리 화면에 접근할 수 있고, 일반 사용자는 차단된다.
- 관리자가 승인/거절(사유)하면 신청자의 상태·권한이 즉시 갱신되고 앱 내 통지가 표시된다.
- 승인된 사용자는 첫 진입(M2 온보딩) 경로로 이동할 수 있다.
- 동일 연락처 중복 신청 시 새 신청을 만들지 않고 기존 상태를 안내한다.

### Dependencies
- 선행: 없음(스캐폴드 위에서 시작). Supabase 프로젝트 연결·환경 변수 설정 필요.
- 후행: M2~M7은 모두 M1의 인증·게이트·디자인 시스템을 전제로 한다.

## Milestone 2 - 온보딩 + 아침 브리핑 MVP (개장 전 방어 브리핑)

### Goal
승인된 사용자가 주요 보유 종목을 입력하고, 오늘의 행동 결론(방어/유지/축소/금지)과 근거를 3분 안에 확인하는 핵심 가치 루프를 완성한다.

### User Value
출근 직후, 흩어진 시장 신호 대신 "오늘 내 종목 기준 행동 결론"을 결론부터 받아본다. 서비스의 aha 모먼트.

### Scope
- 환영/빠른 등록 화면: 종목명·비중·레버리지 여부로 최대 10개 입력(필수 최소 입력).
- 아침 브리핑 대시보드: AI 종합 판단(결론 우선), 9단계 장세, 하락 리스크 점수, 해야 할/하지 말아야 할 행동 라벨, 버핏식 보조 관점.
- 근거 상세 화면(기본): 금리·환율·유가·미국장·반도체·선물·섹터 등 근거 점수와 핵심 이슈.
- 리스크 코칭 카피 톤(브랜드 보이스) 적용: 단정형 투자 명령·공포 표현 배제.

### Out of Scope
- 2차원 맵·수급·이슈 지속성 등 심화 근거(M6).
- 종목별 위험선·매매 원칙·알림(M3).
- 장중·마감 흐름(M4, M5).

### Functional Requirements Covered
FR-001, FR-002, FR-003, FR-004, FR-005, FR-029, FR-006(버핏식 보조 관점, P1)

### Completion Criteria
- 승인된 신규 사용자가 종목 입력부터 행동 결론 확인까지 3분 내 도달할 수 있다(SC-001).
- 모든 행동 결론에 최소 1개 이상의 관찰 가능한 근거가 붙는다(SC-002).
- 종목 0개 등 빈 상태에서 최소 입력 유도 안내가 동작한다.
- 모든 문구가 리스크 코칭 언어로 작성된다(단정형 명령 없음).

### Dependencies
- 선행: M1(인증·게이트·디자인 시스템).
- 데이터 의존: 시장 데이터·AI 판단 로직(오픈 이슈) — Notes의 데이터 공급 가정 적용.

## Milestone 3 - 포지션 관리·위험선·매매 원칙·알림 채널 (포지션 등록과 원칙 설정)

### Goal
사용자가 실제 위험 노출(비중·손익률·레버리지·섹터)을 정리하고, 종목별 추천 위험선·개인 매매 원칙·알림 채널을 확정한다.

### User Value
막연한 보유 목록이 "장세·가격 조건에 따라 행동 원칙이 붙은 관리 대상"으로 바뀐다.

### Scope
- 포지션 관리 화면: 비중·현재가·손익률·레버리지·투자 기간 보완, 종목별 위험도 판단.
- 섹터 추천·판정(반도체·2차전지 등)과 섹터별 장세 단계·근거(P1).
- 위험선/알림 설정 화면: 전저점·이동평균·급락률·시장 연동 기반 추천 위험선 + 사용자 지정 가격선.
- 매매 원칙 화면: 레버리지 신규 매수 금지·물타기 금지 등 원칙 선택·저장.
- 알림 연락처 설정: 휴대폰·이메일 입력, 인증·수신 동의(외부 발송 채널 기반 마련).

### Out of Scope
- 실제 장중 알림 발송 트리거(M4).
- 마감 전 판단(M5).

### Functional Requirements Covered
FR-007, FR-009, FR-010, FR-011, FR-008(섹터 판정, P1)

### Completion Criteria
- 종목별 위험선이 적용되고, 개인 원칙이 저장되며, 알림 채널 1개 이상이 인증·동의 완료된다(Scenario 2 완료 조건).
- 알림 인증 미완료 시 외부 알림 대신 앱 내 알림으로 폴백한다(Edge Case).

### Dependencies
- 선행: M1, M2(종목 입력 기반).
- 후행: M4의 장중 알림은 본 마일스톤의 위험선·원칙·알림 채널을 사용한다.

## Milestone 4 - 장중 리스크 알림·체결 반영 (장중 모니터링)

### Goal
장중 위험 조건이 닿으면 원칙 기반 알림을 보내고, 사용자가 장중 체결을 입력하면 최신 포지션으로 재판단하며, 하루 몇 차례 짧은 시황을 전달한다.

### User Value
화면을 계속 보지 못해도 위험 변화를 놓치지 않고, 충동 매매 대신 저장한 원칙으로 돌아온다.

### Scope
- 장중 알림 화면: 행동·금지·이유 + 미준수 리스크(`원인 → 취약 종목 → 손실 결과 → 확인할 지표`).
- 알림 우선순위: 레버리지·급락·금리 급등·섹터 급락을 더 강하게 전달(P1).
- 장중 포지션 변경 화면: 매도·추가 매수·신규 입력 후 위험도·행동 지침 재계산.
- 장중 시황 요약 화면: 하루 3회 수준의 시장 단계·내 종목 주의·피할 행동(P1).

### Out of Scope
- 마감 전 다음날 판단(M5).
- 증권 계좌 자동 연동(비범위).

### Functional Requirements Covered
FR-012, FR-013, FR-015, FR-014(P1), FR-016(P1)

### Completion Criteria
- 장중 위험 알림이 항상 "행동·금지·이유" 3요소를 포함하고, 미준수 리스크가 있으면 정해진 순서를 지킨다(SC-003).
- 장중 체결 입력 시 최신 포지션 기준으로 위험도·행동 지침이 다시 계산·표시된다(SC-004).

### Dependencies
- 선행: M1, M2, M3(위험선·원칙·알림 채널).
- 데이터 의존: 장중 시세·지표 스트림(오픈 이슈).

## Milestone 5 - 마감 전 다음날 판단·야간 이벤트 (마감 30분 전)

### Goal
장마감 30분 전, 오늘 흐름·야간 선물·금리·유가·환율·수급·보유 종목을 반영해 종목별 유지/축소/매도/현금화 판단을 제시하고, 야간 미국 지표·실적을 자동 스캔해 결과별 시나리오를 발표 전에 보여준다.

### User Value
다음날 갭출발로 기회·위험을 놓치기 전에, 마감 전에 미리 대응 판단을 받는다.

### Scope
- 마감 전 판단 화면: 종목별 판단, 갭 위험·권장 대응, 근거, [내일 아침 다시 보기 예약].
- 야간 이벤트 자동 스캔: 미국 경제지표·연준 일정·주요 기업(반도체·AI 인프라) 실적(P1).
- 발표 전 결과별 시나리오(상회/부합/하회/실적 부진 등)와 예상 범위(P1).

### Out of Scope
- 심화 분석(2차원 맵·수급·이슈 지속성)은 M6에서 공용 모듈로 제공.
- 행동 기록·복기(M7).

### Functional Requirements Covered
FR-017, FR-018(P1), FR-019(P1)

### Completion Criteria
- 마감 전 화면에서 종목별 판단(유지/축소/매도/현금화)과 갭 위험·권장 대응이 제시된다.
- 야간 주요 지표·실적이 사용자가 묻지 않아도 자동 포함되고, 결과별 시나리오가 발표 전에 표시된다.

### Dependencies
- 선행: M1, M2, M3.
- 데이터 의존: 야간 시세·경제/실적 캘린더(오픈 이슈).

## Milestone 6 - 시장·섹터 분석 심화 (근거 강화)

### Goal
브리핑과 마감 전 판단의 근거를 강화하는 공용 분석 모듈을 제공한다: 큰 장세×오늘 상황 2차원 맵, 외국인·기관·개인 수급과 자금 이동, 이슈 지속성.

### User Value
결론의 신뢰도를 높여, 단순 호재·악재가 아니라 "큰 흐름 안의 현재 위치"로 위험을 이해한다.

### Scope
- 2차원 맵: `큰 장세 압력 낮음↔높음` × `오늘 상황 완화↔악화` 위치 판정.
- 수급: 코스피·종목별 외국인·기관·개인 수급, 거래량, 프로그램 매매, 대형 IPO·테마 자금 흡수, 채권으로의 방어적 자금 이동.
- 이슈 지속성: 하루짜리 소음/며칠 이상 리스크/특정 섹터 한정 구분.
- 위 모듈을 아침 브리핑 근거 상세와 마감 전 판단 화면에 공통 노출.

### Out of Scope
- 결론·행동 라벨 자체(M2)와 마감 전 종목별 판단(M5)은 이미 존재. 본 마일스톤은 근거 심화만 추가.

### Functional Requirements Covered
FR-020, FR-021, FR-022 (모두 P1)

### Completion Criteria
- 브리핑·마감 전 화면에서 2차원 맵·수급·이슈 지속성 근거를 함께 확인할 수 있다.
- 호재/악재 충돌 시 단일 단정 대신 장세 민감도를 곱한 종합 결론으로 표시된다(Edge Case).

### Dependencies
- 선행: M2(브리핑), M5(마감 전). 두 화면이 있어야 근거 모듈을 붙일 수 있다.
- 데이터 의존: 수급·채권·자금 이동 데이터(오픈 이슈).

## Milestone 7 - 행동 기록·판단 갭·개인화 인사이트 (복기와 학습)

### Goal
사용자가 날짜·종목별로 안내·실제 행동·결과를 기록하고, 누적 데이터로 판단 갭과 반복 패턴을 복기하며, 동의 기반 개인화 인사이트를 제공한다.

### User Value
기억에 의존한 후회 대신, 실제 기록으로 다음 의사결정 기준을 개선한다.

### Scope
- 행동 기록 입력 화면: 안내·실제 행동·따름 여부·이유·결과(당일·다음날·3거래일·1주일).
- 판단 갭 리포트 화면: 준수 비율, 미준수 후 수익/손실 사례 분리, 반복 패턴(P1).
- 유사 상황 회상 화면: 현재와 겹치는 조건으로 과거 사례 매칭(P1).
- (이후) 오판 분석 리포트, 개인화 인사이트, 개인화 학습 제어(켜기/끄기·삭제·초기화)(P2).

### Out of Scope
- 서비스 전체 모델 학습·외부 활용(동의 없는 사용 금지, 비범위).

### Functional Requirements Covered
FR-023, FR-024(P1), FR-025(P1), FR-026(P2), FR-027(P2), FR-028(P2)

### Completion Criteria
- 사용자가 1건 이상 기록을 남기면 판단 갭 리포트에서 준수 비율과 반복 패턴 요약을 확인할 수 있다(SC-005).
- 개인화 학습 동의를 끄면 과거 기록을 다음 안내에 반영하지 않고, 삭제·초기화 요청을 즉시 반영한다(Edge Case).

### Dependencies
- 선행: M1, M2(안내 생성), M3~M5(기록 대상 안내 다양화).
- P2 항목(FR-026~028)은 개인정보·동의 설계 선행 필요(오픈 이슈).

## Notes

- **데이터 소스 미정(최대 리스크):** 실시간 시세·매크로 지표·뉴스·미국 경제/실적 캘린더의 공급처·갱신 주기·라이선스가 PRD 오픈 이슈로 미정이다. M2·M4·M5·M6의 다수 FR이 여기에 의존한다. 데이터 계약이 정해지기 전에는 각 마일스톤을 "데이터 공급 가정" 위에서 UI·입력·상태·표현 계층 우선으로 진행하고, 실제 데이터 연동은 별도 작업으로 분리할 것을 권장한다.
- **AI 판단 비용/지연:** 종합 판단을 LLM으로 생성할 경우 응답 지연·비용이 "3분 내 결론(SC-001)"과 장중 실시간성에 영향을 줄 수 있다. 모델·캐싱 전략은 M2 착수 전 검토 필요(가정).
- **관리자 인증·권한 관리:** 최초 관리자 시드, 역할 관리, 신청 처리 감사 로그 정책이 미정이다(M1 범위 내 기본 동작은 구현하되 운영 정책은 오픈 이슈).
- **신청 본인확인·개인정보·재신청 정책:** 이용 신청의 본인 인증 수준, 개인정보 수집·보관·삭제, 거절 후 재신청·권한 만료 정책이 미정이다. M1은 기본 동작으로 진행하고 정교한 정책은 후속 결정으로 둔다. 알림 연락처 인증(FR-011, M3)과 신청 단계 연락처의 중복·정합 정리가 필요하다.
- **표현 경계(법적 리스크):** "투자 조언처럼 보이는 단정"과 "원칙 기반 리스크 코칭"의 경계, 면책·고지 문구 정책을 카피·구현 단계에서 일관되게 지켜야 한다(전 마일스톤 공통).
- **테스트·CI 미설정:** 현재 테스트 러너·CI가 없다. 각 마일스톤의 완료 기준을 검증하려면 최소한의 테스트 전략(가정)을 별도로 합의해야 한다.
- **목업 우선 진행 옵션:** 데이터·AI 의존도가 높으므로, 각 마일스톤을 본 구현 전에 frontend 목업으로 먼저 검증하는 흐름(vibe-build의 mockup-plan/mockup-build)을 권장한다.
