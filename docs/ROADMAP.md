## Current Implementation Snapshot

현재 코드베이스는 **프론트엔드 목업 스프린트가 완료된 상태**다. `docs/mockups/plan.md`에 정의된 18개 화면(S001~S018)이 모두 `app/` 하위 Next.js 라우트로 구현되어 있으며, `app/_data/mock.ts`의 더미 데이터로만 동작한다.

**현재 있는 것:**
- 18개 화면 UI (더미 데이터 기반 정적 목업 + 데모 상태 전환)
- 글로벌 내비 + 서브내비 + 면책 푸터 공통 컴포넌트 (`app/_components/Shell.tsx`)
- 버튼·원시 컴포넌트 (`Button.tsx`, `primitives.tsx`)
- 디자인 시스템 (Guard Blue 토큰·Pretendard·컴포넌트 적용됨)
- Next.js App Router 라우트 전체 구조 (`/apply`, `/briefing`, `/positions`, `/journal` 등 18개 경로)
- Supabase 패키지 설치됨 (클라이언트 초기화·연결 미완료)

**현재 없는 것:**
- 실제 인증·세션·권한 관리 (Supabase Auth)
- DB 스키마·마이그레이션·데이터 저장·조회
- 승인 게이트 (미승인 사용자 라우트 보호)
- 시장 데이터 연동 (시세·매크로·뉴스)
- AI 종합 판단 생성 로직
- 알림 발송 인프라 (SMS·이메일)
- 서버 액션·API 라우트 (백엔드 로직 전무)

## Milestone 1 - 승인 게이트 & 인증

### Goal
Supabase Auth 기반으로 이용 신청 접수, 관리자 검토·승인, 게이트 접근 제어를 실제로 작동시킨다. 승인된 사용자만 핵심 기능에 진입할 수 있는 상태를 만든다.

### User Value
서비스를 처음 알게 된 방문자가 신청을 제출하고, 관리자가 승인하면 그 사용자만 실제로 서비스에 진입할 수 있다. 미승인 상태에서는 어떤 경로로 접근해도 차단된다.

### Scope
- Supabase Auth 연동 및 세션 처리
- `applications` 테이블(신청자 정보·상태·거절 사유), `user_roles` 테이블(admin/user 역할) DB 스키마 설계 및 마이그레이션
- 이용 신청 폼 → DB 저장 서버 액션 (중복 신청 처리 포함)
- 신청 상태 조회 (대기/승인/거절, 사유)
- 관리자 신청 목록 조회·상세 API 및 승인/거절 서버 액션
- 승인/거절 시 신청자 `user_status` 즉시 갱신
- Next.js 미들웨어 기반 라우트 보호: 미승인 사용자의 `/briefing`, `/positions`, `/journal` 등 접근 → `/apply/status`로 리다이렉트
- `/admin/*` 경로 관리자 인증 게이트
- 최초 관리자 시드 방법 확정 및 구현

### Out of Scope
- 카카오·SMS·이메일 결과 통지 (M4에서 발송 인프라와 함께 구현)
- 소셜 로그인, PASS 수준 본인 인증
- 재신청·만료 정책 (PRD 오픈 이슈)

### Functional Requirements Covered
FR-030, FR-031, FR-032, FR-033, FR-034, FR-036

### Completion Criteria
- 신규 방문자가 신청을 제출하면 DB에 저장되고 `/apply/status`에서 "대기 중" 상태가 실제로 표시된다.
- 관리자가 `/admin/applications`에서 승인하면 해당 사용자가 로그인 시 `/briefing`에 진입할 수 있다.
- `pending`·`rejected` 상태 사용자가 `/briefing` URL을 직접 입력하면 `/apply/status`로 리다이렉트된다.
- 일반 사용자 세션으로 `/admin/*`에 접근하면 권한 없음 화면이 표시된다.
- 동일 연락처 중복 신청 시 기존 상태가 안내된다.

### Dependencies
없음 (첫 번째 마일스톤)

## Milestone 2 - 포지션·원칙·설정 저장

### Goal
승인된 사용자가 첫 진입에서 입력한 보유 종목, 위험선, 개인 매매 원칙, 알림 채널을 DB에 저장하고, 이후 판단의 개인화 입력으로 활용할 수 있게 한다.

### User Value
처음 접속한 승인 사용자가 주요 종목·비중·레버리지를 입력하고 위험선·원칙을 설정하면, 다음 브리핑·알림이 자신의 포지션과 원칙을 실제로 반영한다.

### Scope
- `positions`, `principles`, `alert_channels` DB 테이블 설계 및 마이그레이션
- 종목 추가·수정·삭제 서버 액션 (최대 10개 제한, 중복 처리)
- 섹터 자동 추천 로직 (종목명 기반 룩업 테이블로 시작)
- 종목별 위험도 계산 (레버리지·손익률·비중 기반 규칙)
- 추천 위험선 계산 로직 (이동평균·급락률·비중 기반 규칙 — 실제 시세는 M3 이후 고도화)
- 개인 원칙 선택·저장·조회 서버 액션
- 알림 채널 입력 및 OTP 인증 흐름 (발송 인프라는 M4와 공유, M2에서 기반 마련)
- 장중 체결 입력 후 포지션 갱신 서버 액션 (재판단 트리거는 M4에서 구현)

### Out of Scope
- 실제 시세 기반 위험선 자동 업데이트 (M3 이후 고도화)
- 증권사 계좌 자동 연동 (비범위)
- 알림 실제 발송 (M4에서 구현)

### Functional Requirements Covered
FR-001, FR-007, FR-008, FR-009, FR-010, FR-011

### Completion Criteria
- 승인된 사용자가 `/onboarding`에서 종목을 입력·저장하면 DB에 기록되고 `/positions`에서 다시 조회할 수 있다.
- 원칙 선택 후 `[저장하기]`를 누르면 DB에 반영되어 다음 접속 시 동일 선택 상태가 유지된다.
- 추천 위험선이 계산되어 화면에 표시되고, `[적용하기]` 후 해당 종목의 위험선이 저장된다.
- 알림 채널 OTP 인증 완료 시 `alert_channels.verified = true`로 저장된다.
- 인증 미완료 상태에서는 외부 알림 대신 앱 내 알림 폴백 안내가 표시된다.

### Dependencies
M1 (사용자 인증 세션 및 user_id 필요)

## Milestone 3 - 아침 브리핑 (데이터 + AI 판단)

### Goal
시장 데이터를 실제로 불러와 AI가 9단계 장세를 판정하고 종합 판단·근거 점수를 생성하여 아침 브리핑 화면에 실제 데이터로 표시한다.

### User Value
출근 직후 접속한 사용자가 더미 데이터가 아닌 오늘의 실제 시장 상황과 자신의 포지션을 반영한 행동 결론을 3분 안에 확인할 수 있다.

### Scope
- 시장 데이터 수집 파이프라인: 외부 데이터 소스 연동 (미국장·환율·금리·유가·반도체 지수·코스피 선물 등 — Notes 1번 참조)
- 데이터 캐싱 전략 (배치 선생성 + 캐싱으로 SC-001 3분 이내 응답 보장)
- AI 종합 판단 생성: LLM 기반 결론·행동/금지 라벨·버핏식 관점 생성 (Notes 2번 참조)
- 9단계 장세 판정 로직 (0~100 리스크 점수 + 단계 라벨)
- 근거 점수 산출 (금리·환율·유가·수급·채권·섹터별 위험도)
- 이슈 지속성 분류 (하루 소음 / 며칠 리스크)
- 2차원 맵 데이터 (큰 장세 압력 × 오늘 상황 좌표 계산)
- 수급 데이터 연동 (외국인·기관·개인·채권·프로그램 매매)
- `briefing_snapshots` 테이블 저장 (날짜별 판단 기록 — M5 행동 기록과 연결)
- 사용자 포지션·섹터 기반 개인화 반영
- FR-029 리스크 코칭 언어 시스템 프롬프트 설계 및 적용
- 폴백 처리: 데이터 실패 시 저장된 기준 가격 대체 + "확인 불가" 라벨

### Out of Scope
- 장중 실시간 데이터 갱신 (M4에서 구현)
- 야간 이벤트 스캔 (M4에서 구현)

### Functional Requirements Covered
FR-002, FR-003, FR-004, FR-005, FR-006, FR-020, FR-021, FR-022, FR-029

### Completion Criteria
- 브리핑 화면이 당일 실제 시장 데이터 기반으로 생성된 종합 판단(행동 라벨 포함)을 표시한다.
- 근거 점수가 실제 지표에서 계산된 값으로 표시된다.
- 모든 브리핑 텍스트가 투자 명령이 아닌 리스크 코칭 언어로 생성된다 (FR-029 준수).
- 시장 데이터 일부 실패 시 해당 근거만 "확인 불가"로 표시되고 나머지 판단은 유지된다.
- SC-001: 첫 접속에서 판단 확인까지 3분 이내.

### Dependencies
M1 (인증), M2 (사용자 포지션·섹터 데이터)

## Milestone 4 - 장중 알림 & 마감 전 판단

### Goal
장중 실시간 시장 변화를 모니터링하고, 위험 조건이 충족되면 원칙 기반 알림을 사용자에게 실제로 발송한다. 장중 체결 반영 후 재판단, 마감 전 다음날 시나리오 판단까지 완성한다.

### User Value
장중에 화면을 보지 못하는 사용자가 위험 신호를 문자·이메일로 받고, 마감 전 다음날 갭 위험과 대응 판단을 확인해 충동 매매 대신 저장한 원칙으로 돌아올 수 있다.

### Scope
- 장중 시세·지표 폴링 또는 웹소켓 연동 (위험 조건 감지)
- 알림 트리거 로직: 종목별 위험선 도달·급락·금리 급등·섹터 급락 조건 평가
- 위험 강도 우선순위 분류 (FR-014: 레버리지·급락 조건 최우선)
- SMS·이메일 발송 서비스 연동 (Notes 3번 참조)
- 알림 내용 생성: 행동·금지·이유·미준수 리스크(원인→취약→손실→지표) + 저장된 원칙 연결
- 하루 3회 장중 시황 요약 발송 스케줄
- 장중 체결 입력 → 포지션 갱신 → AI 재판단 실행 (M3 판단 로직 재호출)
- 승인·거절 결과 신청자 외부 통지 (M1에서 미룬 FR-035 구현)
- 마감 전 판단 생성: 야간 지표·실적·연준 일정 자동 스캔 + 결과별 시나리오
- 마감 전 종목별 판단 (유지/축소/매도/현금화) 생성
- `alerts` 테이블 저장 (발송 이력·수신 확인·미준수 리스크)

### Out of Scope
- 카카오 알림톡 등 채널 확장 (비범위)
- 자동 주문·매매 실행 (비범위)

### Functional Requirements Covered
FR-012, FR-013, FR-014, FR-015, FR-016, FR-017, FR-018, FR-019, FR-035

### Completion Criteria
- 사용자 종목이 위험선에 도달하면 10분 이내 알림이 인증된 채널(SMS 또는 이메일)로 발송된다.
- 알림 내용에 행동·금지·이유 3요소가 항상 포함되고, 미준수 리스크가 `원인 → 취약 종목 → 손실 결과 → 확인할 지표` 순서로 표시된다 (SC-003).
- 장중 체결 입력 후 `[최신 상태로 다시 판단]`을 누르면 브리핑 화면이 갱신된 포지션 기준으로 재계산된다 (SC-004).
- 마감 전 화면에 야간 이벤트 자동 스캔 결과와 종목별 판단이 실제 데이터로 표시된다.
- 승인·거절 결과 통지가 신청자에게 전달된다 (FR-035).

### Dependencies
M1 (인증·권한), M2 (포지션·원칙·알림 채널 설정), M3 (판단 생성 로직)

## Milestone 5 - 행동 기록 & 갭 리포트

### Goal
사용자가 날짜·종목별로 스탁가드 안내와 실제 행동·결과를 DB에 저장하면, 준수 비율과 반복 패턴을 판단 갭 리포트로 실제 데이터 기반으로 확인할 수 있게 한다. 유사 상황 회상도 실제 기록 데이터 기반으로 작동한다.

### User Value
매매가 지나간 뒤 패턴을 돌아보려는 사용자가 안내를 얼마나 따랐는지, 따르지 않았을 때 결과가 어땠는지를 객관적으로 확인할 수 있다.

### Scope
- `action_logs` 테이블 설계·마이그레이션 (날짜·종목·안내 내용·실제 행동·따름 여부·이유·결과)
- 행동 기록 입력 서버 액션 (저장·수정·조회)
- 결과 타임라인 저장 (당일·다음날·3거래일·1주일 가격/손익)
- 판단 갭 리포트 계산 로직 (준수 비율, 미준수 후 수익·손실 사례 분리)
- 소표본 신뢰도 경고 (N건 미만 시 "표본이 적어 신뢰도 제한" 안내)
- 반복 패턴 추출 로직 (동일 장세·종목·행동 조합 집계)
- 유사 상황 회상 매칭 로직 (현재 장세·종목·섹터 조건과 과거 `action_logs` 매칭)
- `briefing_snapshots`와 `action_logs` 조인 (M3 판단 내용과 연결)

### Out of Scope
- 오판 분석 리포트 (M6에서 구현)
- 개인화 인사이트 (M6에서 구현)

### Functional Requirements Covered
FR-023, FR-024, FR-025

### Completion Criteria
- 사용자가 1건 이상 행동 기록을 저장하면 갭 리포트 화면에 준수 비율과 미준수 사례가 실제 데이터로 표시된다 (SC-005).
- 기록이 기준 건수 미만이면 소표본 신뢰도 경고가 함께 표시된다.
- 유사 상황 회상 화면이 현재 장세 조건과 과거 기록을 매칭해 관련 사례를 표시한다.

### Dependencies
M1 (인증), M3 (브리핑 스냅샷 데이터)

## Milestone 6 - 개인화 인사이트 & 오판 분석 (P2)

### Goal
누적된 행동 기록과 브리핑 판단 이력을 기반으로 AI가 개인화 인사이트와 오판 분석 리포트를 생성한다. 사용자가 개인화 학습 반영 여부를 제어하고 기록을 삭제·초기화할 수 있다.

### User Value
패턴을 개선하려는 사용자가 자신의 반복 실수·강점 조건·취약 장세를 AI 요약으로 확인하고, 스탁가드가 틀렸을 때 그 원인까지 함께 돌아볼 수 있다.

### Scope
- 오판 분석 리포트 생성 (당시 근거 + 이후 바뀐 변수 분리 + 다음 반영안 — LLM 생성)
- 개인화 인사이트 생성 (반복 실수·잘 맞는 판단·취약 장세 요약 — LLM 생성)
- 개인화 반영 동의 ON/OFF 저장 및 반영 여부 제어 로직
- 특정 기록 제외·전체 삭제·학습 초기화 서버 액션
- "내일 아침 다시 보기" 예약 저장 → 다음날 브리핑 접속 시 예약 알림 표시
- `personalization_settings` 테이블 (동의 상태·제외 기록 ID 목록)

### Out of Scope
- 서비스 전체 모델 파인튜닝 파이프라인 (비범위)
- 외부 활용·공유 기능 (비범위)
- 버핏 외 투자 대가 보조 관점 (비범위)

### Functional Requirements Covered
FR-026, FR-027, FR-028, FR-038

### Completion Criteria
- 오판 분석 리포트가 당시 근거와 이후 바뀐 변수를 분리해 실제 AI 생성 텍스트로 표시된다.
- 개인화 반영 동의를 OFF로 설정하면 이후 인사이트 생성에 해당 사용자의 기록이 포함되지 않는다.
- 기록 전체 삭제·초기화 요청이 즉시 반영되어 갭 리포트와 인사이트에 빈 상태가 표시된다.
- "내일 아침 다시 보기" 예약 후 다음날 브리핑 접속 시 예약 알림이 표시된다.

### Dependencies
M1 (인증), M3 (브리핑 판단 이력), M5 (행동 기록·갭 리포트 데이터)

## Notes

1. **시장 데이터 소스 선정 필수 (M3 블로커):** 시세·매크로·뉴스·실적 캘린더의 외부 API를 M3 시작 전에 확정해야 한다. 9단계 장세 판정의 정확도와 SC-001(3분 이내 응답)이 여기에 직접 좌우된다. 데이터 라이선스·갱신 주기·비용도 함께 검토할 것.

2. **LLM 모델·캐싱 전략 (M3 블로커):** AI 종합 판단 생성 시 응답 지연과 비용이 SC-001과 충돌할 수 있다. 아침 브리핑은 배치 선생성(전날 밤 준비) + 캐싱, 장중 재판단은 실시간 경량 호출 분리 방식을 권장한다. 모델 선정은 PRD에 미정이므로 M3 착수 전에 결정 필요.

3. **SMS·이메일 발송 서비스 (M2·M4 블로커):** 알림 채널 OTP 인증(FR-011, M2)과 실제 알림 발송(FR-012·035, M4)이 같은 인프라를 공유한다. M2 시작 전에 발송 서비스(Twilio, AWS SES/SNS 등)를 확정해야 채널 인증과 발송 구현이 함께 진행될 수 있다.

4. **관리자 시드 계정 방법 확정 (M1 선행 결정):** M1에서 최초 관리자 시드 방법(Supabase 대시보드에서 직접 role 부여 또는 별도 스크립트)을 명확히 정해야 한다. 일반 신청 흐름으로 관리자 권한 획득 불가 원칙(PRD 11절 Assumptions)을 코드로 보장한다.

5. **FR-035(통지 발송) M4 배치 이유:** 승인·거절 통지는 SMS/이메일 발송 인프라가 필요하고, 이 인프라가 M4 장중 알림 구현과 공유된다. M1에서 UI 상태 반영(FR-034)을 먼저 구현하고, 외부 통지는 M4에서 발송 인프라와 함께 추가한다.

6. **P2 기능(M6) 착수 타이밍:** FR-026~028, FR-038은 개인정보·동의 처리 설계와 누적 기록이 선행되어야 실질적 가치가 생긴다. M1~M5 완료 후 착수를 권장한다.

7. **테스트·CI 미설정:** 현재 테스트 러너·CI가 없다. 각 마일스톤의 완료 기준을 검증하려면 최소한의 테스트 전략을 마일스톤 착수 전에 합의해야 한다.
