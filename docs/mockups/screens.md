# Mockup Screens

## Overview

스탁가드 v1 목업 18개 화면의 상세 설계다. 각 화면은 목적·진입/이탈·핵심 행동·ASCII 레이아웃·UI 영역·카피·더미 데이터·상태·반응형·리뷰 질문·확정 기준 순으로 정리한다. 화면 ID는 `docs/mockups/plan.md`와 일치하며(S001~S018), 흐름 순서대로 배치했다. ASCII 도식은 정교한 와이어프레임이 아니라 정보 구조를 빠르게 이해하기 위한 스케치다.

## Global Mockup Assumptions

- **레이아웃 셸:** 상단 얇은 글로벌 내비(`global-nav`, 검정 44px) + 서피스별 서브내비(`sub-nav-frosted`, 패치먼트 52px, 좌측 화면명/우측 CTA). 관리자 화면(S003)은 `[관리자]` 배지로 구분.
- **시각 시스템(DESIGN.md):** Guard Blue(#0066cc) 단일 액센트, 결론은 라이트/다크 결정 타일에 크게, 근거는 그 아래. pill CTA(`button-primary`), 유틸리티 카드는 `rounded.lg`. 차트/포지션 아티팩트에만 단일 섀도. 이모지·그라데이션 금지.
- **타이포:** Pretendard. 결론 헤드라인 `display-lg`(40px/600), 본문 `body`(17px/400/1.47).
- **보이스:** 결론 먼저 → 근거 → 행동 원칙. `~하세요/~입니다/~할 수 있습니다`. 금지어(무조건 매도/매수, 대박/몰빵, 패닉/붕괴, FOMO, 단정형 명령) 사용하지 않음.
- **더미 데이터:** 종목은 삼성전자·SOXL·SK하이닉스·한화에어로 등 고정 샘플. 날짜는 2026-06-14 기준. 점수는 0~100, 장세는 "변동장 3단계" 등 고정 라벨. 모든 시세·점수는 목업용 더미.
- **반응형 기준:** 모바일(≤640px, 단일 컬럼) / 데스크톱(≥1069px, 타일·그리드). 결정 타일은 834px에서 2→1 컬럼.
- **게이트:** 미승인 사용자가 S004~S018 진입을 시도하면 S002로 안전하게 차단(목업에서는 차단 진입 상태로 표현).

## Screen S001 - 이용 신청 화면

**Purpose:** 이용 권한이 없는 방문자가 최소 정보로 이용을 신청하고 검토를 요청한다.

**Entry Points:**
- 랜딩/홍보 링크로 처음 방문
- 미승인 상태에서 핵심 기능 진입 시도 → 신청 유도

**Exit Points:**
- `[신청하기]` → S002(승인 대기/상태)
- 동일 연락처 기존 신청 → S002로 기존 상태 안내

**Primary User Action:** `[신청하기]` 제출

**Secondary Actions:**
- 개인정보·이용 동의 체크
- 투자 경험·신청 동기 입력(선택)

**Layout ASCII:**

```text
+--------------------------------------------------+
| 스탁가드                              [로그인]   |
+--------------------------------------------------+
| 스탁가드 이용 신청                               |
| 승인 후 개인화 리스크 코칭을 이용할 수 있습니다  |
+--------------------------------------------------+
| 이름      [ 홍길동                        ]      |
| 이메일    [ hong@example.com              ]      |
| 휴대폰    [ 010-0000-0000                 ]      |
| 투자 경험 [ 3년 / 국내·미국장             ]      |
| 신청 동기 [ 변동장 대응이 어려워서         ]      |
|                                                  |
| [ ] 개인정보 수집·이용에 동의 (필수)             |
+--------------------------------------------------+
|                              [   신청하기   ]    |
+--------------------------------------------------+
```

**Key UI Regions:**
- Header: 글로벌 내비 + 로그인(기존 신청자용)
- Main: 신청 폼(필수: 이름·연락처·동의 / 선택: 경험·동기)
- Actions: `[신청하기]` 단일 pill CTA(동의 전 비활성)
- Supporting Area: 승인 후 이용 가능 안내 문구

**Content / Copy Notes:**
- 헤드라인: "스탁가드 이용 신청" / 서브: "승인 후 개인화 리스크 코칭을 이용할 수 있습니다."
- 동의 라벨 옆 "(필수)" 표기. 경험·동기는 "검토 참고용" 보조 문구.

**Data / Placeholder Needs:**
- 샘플: 홍길동 / hong@example.com / 010-0000-0000 / "3년 / 국내·미국장" / "변동장 대응이 어려워서"

**States:**
- Default: 빈 폼, `[신청하기]` 비활성(동의 미체크)
- Empty: 동일(첫 진입)
- Loading: 제출 중 — 버튼 "신청 중…", 입력 잠금
- Error: 동일 연락처 기존 신청 — "이미 신청하신 연락처입니다. 현재 상태를 확인하세요." → S002 안내
- Selected / Active: 동의 체크 시 `[신청하기]` 활성

**Responsive Notes:**
- Mobile: 단일 컬럼, 입력 풀너비, CTA 하단 고정
- Desktop: 중앙 ~640px 폼, CTA 우측 정렬

**Review Questions:**
- [ ] 최소 입력(이름·연락처·동의)이 부담 없이 느껴지는가?
- [ ] 필수/선택 입력 구분이 명확한가?
- [ ] 동일 연락처 재신청 안내가 자연스러운가?

**Acceptance Criteria:**
- [ ] 필수/선택 입력과 동의 체크가 시각적으로 구분된다
- [ ] `[신청하기]` CTA가 명확하다
- [ ] 제출 후 S002로 가는 흐름이 설명 가능하다

**Notes:**

본인 인증 수준·신청 정보 보관 정책은 오픈 이슈. 목업에서는 입력·제출 UI만 표현한다.

## Screen S002 - 승인 대기/상태 화면

**Purpose:** 승인 전에는 핵심 기능 대신 현재 신청 상태를 명확히 보여줘 사용자가 방치되지 않게 한다.

**Exit Points:**
- (승인 시) `[시작하기]` → S004

**Primary User Action:** (승인 시) `[시작하기]` / (대기 시) `[상태 새로고침]`

**Secondary Actions:**
- 통지 채널 확인/보완

```text
+--------------------------------------------------+
| 스탁가드                                         |
+--------------------------------------------------+
|   ● 신청 상태: 대기 중                           |
|                                                  |
|   신청이 접수되었습니다.                         |
|   관리자 검토 후 결과를 알려드립니다.            |

|   접수일     2026-06-14                          |
|   통지 채널  이메일                              |
+--------------------------------------------------+
|   [ 상태 새로고침 ]                              |
+--------------------------------------------------+

승인 상태:                       거절 상태:
+----------------------------+   +----------------------------+
| ● 승인되었습니다           |   | ● 신청이 거절되었습니다    |
```

- Supporting Area: 거절 사유, 다음 안내

**Content / Copy Notes:**

- 거절: "신청이 거절되었습니다. 사유: …" (사용자를 탓하지 않는 톤)
- 통지 미인증: "이메일 인증이 완료되지 않아 앱 내 알림으로 안내합니다."

**Data / Placeholder Needs:**

**States:**
- Default: 대기 중
- Empty: 해당 없음
- Loading: 상태 새로고침 중 — 스피너
- Error: 통지 채널 미인증 안내 / 미승인 직접 접근 차단 진입("승인 후 이용할 수 있습니다")
- Selected / Active: 승인 → `[시작하기]` 노출 / 거절 → 사유 노출

**Responsive Notes:**
- Mobile: 단일 컬럼 카드, 상태 배지 상단
- Desktop: 중앙 정렬 카드 ~520px

**Review Questions:**

- [ ] 거절 사유·다음 안내가 사용자를 탓하지 않는가?
- [ ] 승인 시 게이트 해제(시작하기)가 분명한가?

**Acceptance Criteria:**
- [ ] 세 상태가 모두 시각적으로 표현된다

**Secondary Actions:**

- 신청 상세 펼치기

**Layout ASCII:**

```text
+--------------------------------------------------+

| 홍길동     hong@..       06-14                   |
```

- Header: 화면명 + `[관리자]` 배지

**States:**
- Default: 대기 목록(우선 표시)

- Loading: 목록 로딩
- Error: 일반 사용자 접근 — "관리자 권한이 필요합니다." 차단
- Selected / Active: 거절 ▾ 펼침 → 사유 입력 / 처리 완료 행 상태 갱신

**Responsive Notes:**
- Mobile: 카드형 목록(행 → 카드), 액션 버튼 하단
- Desktop: 테이블형 목록, 상세 인라인 펼침

## Screen S004 - 환영/빠른 등록 화면
