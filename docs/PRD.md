## 1. 개요
- **한 줄 정의:** 개인 투자자가 아침·장중·장마감 전에 시장 데이터와 주요 이슈를 종합 판단받아, 보유 종목의 위험을 줄이고 매매 원칙을 지키도록 돕는 투자 리스크 관제 서비스.   ← idea
- **해결하는 문제:** 변동성 장세에서 미국장·매크로·금리·환율·유가·뉴스가 여러 곳에 흩어져 있어, 출근 직후 짧은 시간에 "오늘 내 종목을 줄여야 하는지"를 판단하기 어렵다. 특히 하락·변동장에서 공포로 인한 외면, 손절 회피, 손실 만회성 레버리지·몰빵 같은 패착이 반복된다.   ← idea
- **타깃 사용자:** 오전 8시 30분 출근 직후 전날 미국장과 오늘 한국장 전망을 확인하고 장중 보유 종목 위험을 점검하려는 개인 투자자. 특히 반도체·2차전지·미국 기술주·국내 대형주를 보유했거나, SOXL 같은 레버리지 ETF를 거래하고, 약세장에서 무리한 매매 경험이 있으며, 감정이 아닌 데이터·원칙으로 결정하려는 투자자.   ← idea
- **플랫폼:** 반응형 웹앱 (데스크톱·모바일 모두 지원). 핵심 사용 시간은 오전 8시 30분, 장중, 장마감 30분 전.   ← idea

## 2. 배경 & 근거
- **시장/경쟁 맥락:** Investing.com·TradingView·Koyfin·Stock Alarm·Benzinga Pro·Finviz Elite 등은 실시간 시세, 차트 조건 알림, 빠른 뉴스, 포트폴리오 추적을 각각 잘 제공한다. 그러나 보유 종목·레버리지 여부·손익률·개인 매매 원칙을 함께 반영해 "오늘 무엇을 줄이고 무엇을 하지 말아야 하는지"까지 장세별로 코칭하는 서비스는 빈자리가 크다. 개인 투자자가 손실을 키우는 순간은 "정보가 없어서"보다 "정보를 보고도 장세에 맞게 행동하지 못해서" 발생한다.   ← benchmark
- **우리의 차별 프레이밍:** 스탁가드는 정보 제공 서비스가 아니라 **"장세별 행동 교정 서비스"**로 포지셔닝한다. 전체 시장·섹터를 상승·변동·하락 9단계로 판정하고, 결론을 먼저 말한 뒤 근거 점수·데이터를 붙이며, 보유 종목·비중·레버리지·손익률·개인 원칙을 반영한다. "해야 할 행동"뿐 아니라 "하지 말아야 할 초보자 실수"를 명확히 말하고, 미국장·매크로를 한국장과 내 종목의 리스크 언어로 번역한다.   ← benchmark

## 3. 목표 (Goals)
- 흩어진 시장 신호를 출근 직후 3분 안에 "오늘의 행동 결론(방어/유지/축소/금지)"으로 바꿔 사용자가 곧바로 판단하게 한다.
- 화면을 계속 보지 못하는 장중에도 위험 변화와 마감 전 다음날 위험을 놓치지 않게 해, 충동 매매 대신 저장한 원칙으로 돌아오게 한다.
- 안내·실제 행동·결과를 기록해 사용자의 반복 패착과 강점 패턴을 학습하고, 다음 코칭을 점점 개인화한다.

## 4. User Scenarios   ← userflow

### Scenario 1 — 개장 전 방어 브리핑
- **사용자:** 아침에 처음 접속하는 개인 투자자
- **상황:** 출근 직후, 전날 미국장 결과와 오늘 한국장 영향이 흩어져 있어 불안한 상태
- **목표:** 주요 보유 종목 기준으로 오늘의 장세 단계와 행동 결론을 3분 안에 확인
- **흐름:** 주요 보유 종목 1~10개(종목명·비중·레버리지 여부) 입력 → 아침 브리핑 대시보드에서 종합 판단·행동·금지·버핏식 관점 확인 → 근거 상세에서 점수와 이슈 확인
- **완료 조건:** 사용자가 `방어 우선`/`유지 가능`/`레버리지 부적합` 같은 오늘의 행동 라벨과 최소 1개 이상의 근거를 확인한 상태

### Scenario 2 — 포지션 등록과 원칙 설정
- **사용자:** 자기 포지션과 위험 기준을 정리하려는 사용자
- **상황:** 막연히 종목을 들고 있고, 어디서 위험해지는지·언제 알림 받을지 정해두지 않은 상태
- **목표:** 주요 포지션·위험선·매매 원칙·알림 채널을 확정
- **흐름:** 포지션 보완(비중·손익률·레버리지·섹터) → 추천 위험선 확인·적용 → 개인 매매 원칙 선택·저장 → 문자·이메일 알림 연락처 입력·인증·수신 동의
- **완료 조건:** 종목별 위험선이 적용되고, 개인 원칙이 저장되고, 알림 채널 1개 이상이 인증·동의 완료된 상태

### Scenario 3 — 장중 리스크 알림과 마감 전 판단
- **사용자:** 장중 화면을 계속 보지 못하는 사용자
- **상황:** 급락·반등·뉴스에 흔들리고, 다음날 갭 위험을 점검하지 못한 상태
- **목표:** 위험 신호가 닿았을 때 행동·금지·이유·미준수 리스크를 확인하고, 마감 전 다음날 대비 판단을 받는다
- **흐름:** 장중 위험 알림 수신·확인 → (필요 시) 장중 체결 반영 후 재판단 → 하루 3회 문자 시황 확인 → 마감 전 야간 지표·실적 시나리오와 갭 위험·권장 대응 확인 → 원칙 재확인
- **완료 조건:** 사용자가 알림의 행동·금지·이유와 마감 전 종목별 판단(유지/축소/매도/현금화)을 확인하고, 변경된 포지션이 최신 상태로 반영된 상태

### Scenario 4 — 행동 기록과 인사이트
- **사용자:** 자신의 판단 패턴을 개선하려는 사용자
- **상황:** 매매가 지나간 뒤 기억에 의존해 후회하는 상태
- **목표:** 날짜별 안내·실제 행동·결과를 기록하고, 누적 갭과 반복 패턴을 확인
- **흐름:** 행동 기록 입력(안내·실제 행동·따름 여부·이유·결과) → 판단 갭 리포트 확인 → 유사 상황 회상 확인 → 오판 분석 리포트 확인 → 개인화 인사이트 확인·동의 반영
- **완료 조건:** 최소 1건의 안내·행동·결과 기록이 저장되고, 사용자가 자신의 반복 패턴 요약을 확인한 상태

### Scenario 5 — 이용 신청과 승인 대기
- **사용자:** 서비스를 처음 알게 된 신규 방문자(아직 이용 권한 없음)
- **상황:** 핵심 기능(아침 브리핑·포지션·알림 등)을 쓰려 하지만 승인 전이라 접근할 수 없는 상태
- **목표:** 이용 신청을 제출하고 승인 여부를 확인해, 승인되면 서비스에 진입
- **흐름:** 이용 신청 폼 작성(이름·연락처·투자 경험/신청 동기) → 개인정보·이용 동의 → 제출 → 승인 대기 안내 확인 → (관리자 결정 후) 상태/거절 사유 확인 → 승인 시 첫 진입(종목 입력)·아침 브리핑으로 이동
- **완료 조건:** 신청이 접수되어 "대기 중" 상태가 표시되고, 승인 시 핵심 기능(첫 진입/아침 브리핑)에 접근할 수 있는 상태

### Scenario 6 — 관리자 신청 검토와 승인
- **사용자:** 서비스 운영 관리자
- **상황:** 신규 이용 신청이 접수되어 검토·결정이 필요한 상태
- **목표:** 신청을 검토해 승인 또는 거절하고, 결과를 신청자에게 통지
- **흐름:** 관리자 인증 → 신청 목록(상태별) 조회 → 신청 상세 확인 → 승인 또는 거절(거절 사유 입력) → 신청자 상태·이용 권한 반영 → 결과 통지 발송
- **완료 조건:** 선택한 신청이 승인/거절로 처리되어 신청자의 상태·이용 권한이 갱신되고, 결과 통지가 발송된 상태

## 6. Success Criteria
- SC-001: 신규 사용자가 첫 진입에서 주요 보유 종목 입력부터 오늘의 행동 결론 확인까지를 3분 안에 완료할 수 있다.
- SC-002: 아침 브리핑·장중 알림·마감 전 판단의 모든 행동 결론에 최소 1개 이상의 관찰 가능한 근거(점수·지표·이슈)가 붙는다.
- SC-003: 장중 위험 알림은 "행동·금지·이유" 3요소를 항상 포함하며, 미준수 리스크가 있는 경우 `원인 → 취약 종목 → 손실 결과 → 확인할 지표` 순서를 지킨다.
- SC-004: 사용자가 장중 체결을 입력하면, 최신 포지션을 기준으로 위험도·행동 지침이 다시 계산되어 표시된다.
- SC-005: 사용자가 1건 이상 행동 기록을 남기면, 판단 갭 리포트에서 안내 준수 비율과 반복 패턴 요약을 확인할 수 있다. *(누적 표본이 적을 때의 신뢰도 기준은 오픈 이슈)*
- SC-006: 신규 방문자가 이용 신청을 제출하면 신청이 접수되어 "대기 중" 상태로 확인되며, 승인 전에는 핵심 기능에 접근할 수 없다.
- SC-007: 관리자가 신청을 승인/거절하면, 해당 신청자의 상태·이용 권한과 결과 통지가 반영된다.

## 7. Edge Cases
- **빈 상태(종목 미입력):** 첫 화면에서 종목이 0개면, 입력을 강요하지 않고 "종목명과 비중만 입력해도 첫 판단을 시작할 수 있습니다" 안내로 최소 입력을 유도한다.
- **데이터 부족/로딩:** 시세·지표를 불러오지 못하면, 빈 화면 대신 "현재 시세를 불러오지 못했습니다. 저장된 기준 가격으로 먼저 판단합니다"로 저장된 기준으로 폴백한다.
- **오류 상태:** 외부 데이터 일부 누락 시 해당 근거를 "확인 불가"로 표시하고, 결론의 신뢰도를 낮춰 표현하되 사용자를 탓하지 않는다.
- **알림 인증 미완료:** 휴대폰·이메일 인증/수신 동의가 없으면 외부 알림(SMS·메일)을 발송하지 않고, 앱 내 알림으로만 대체하며 설정 보완을 안내한다.
- **중복/모순 입력:** 같은 종목 중복 등록 시 합산 또는 경고로 처리한다. 근원 CPI 하락 같은 호재와 헤드라인 CPI·유가 상승 같은 악재가 충돌하면, 단일 호재/악재로 단정하지 않고 현재 장세 민감도를 곱해 종합 결론을 낸다.
- **취소/이탈:** 첫 진입에서 알림 연락처 설정은 건너뛸 수 있고, 이후 자연스럽게 보완하게 한다. 마감 전 판단을 안 보고 이탈해도 "내일 아침 다시 보기 예약"으로 다음 접점을 남긴다.
- **권한/개인화 거부:** 개인화 학습 동의를 끄면 과거 기록을 다음 안내에 반영하지 않고, 기록 삭제·초기화 요청을 즉시 반영한다.
- **단기 반등의 오해:** 반등이 나와도 추세 전환인지 단기 반등인지 구분하지 못하는 상황을 대비해, 선물 방향·금리·수급 재확인 전까지는 추가 진입 보류 문구를 우선한다.
- **중복 신청:** 이미 신청·승인된 연락처로 다시 신청하면 새 신청을 만들지 않고 기존 상태(대기/승인/거절)를 안내한다.
- **미승인 접근:** 대기·거절 상태 사용자가 핵심 기능 URL에 직접 접근해도 신청/상태 화면으로 안전하게 차단한다.
- **거절 후 재신청:** 거절된 사용자의 재신청 허용 여부·대기 기간 정책을 적용한다(정책 자체는 오픈 이슈).
- **관리자 권한 오용:** 일반 사용자가 관리자 신청 관리 화면에 접근하면 차단하고, 권한 없음을 안내한다.
- **통지 실패:** 승인·거절 결과의 외부 채널 통지가 실패하면 앱 내 상태 표시로 폴백한다(알림 인증 미완료 처리와 동일 원칙).

## 11. Assumptions
- v1에서 장중 체결과 결과(가격·손익)는 사용자가 직접 입력하는 것을 기본으로 한다(증권 연동은 비범위).
- 시세·지표·뉴스·실적 캘린더 등 시장 데이터는 외부 데이터 소스로부터 공급된다고 가정한다. 구체적 데이터 제공처·실시간성 수준은 별도 결정이 필요하다(오픈 이슈).
- "주요 보유 종목 최대 10개"는 모든 보유 종목이 아니라 의사결정에 큰 영향을 주는 포지션 기준이다.
- 사용자는 한국 주식시장 중심으로 투자하며, 미국장·매크로 영향을 한국장 관점에서 받기를 원한다.
- 개인화 기록은 해당 사용자에게만 쓰는 참고 데이터이며, 서비스 전체 모델 학습·외부 활용은 별도 동의 없이는 하지 않는다.
- 버핏식 관점은 실제 인물의 실시간 판단이 아니라 장기 가치투자 원칙을 현재 장세에 적용한 보조 프레임으로만 제공한다.
- 서비스는 승인 기반 제한 이용(gated access) 모델로 시작하며, 관리자 승인을 받은 사용자만 핵심 기능을 이용한다.
- 관리자 계정·권한은 운영팀이 별도로 부여하며, 일반 이용 신청 흐름으로는 관리자 권한을 얻을 수 없다.
- 이용 신청 시 수집하는 최소 정보는 이름·연락처와 간단한 투자 경험/신청 동기이며, 세부 항목은 운영 과정에서 조정될 수 있다.

## 12. 오픈 이슈 / 리스크
- **데이터 소스 미정:** 실시간 시세, 매크로 지표, 뉴스, 미국 실적·경제 캘린더의 공급처·갱신 주기·라이선스가 정해지지 않았다. 9단계 장세 판정·근거 점수의 정확도가 여기에 좌우된다.
- **표현 경계(법적 리스크):** "투자 조언처럼 보이는 단정"과 "원칙 기반 리스크 코칭"의 경계를 구현·카피 단계에서 명확히 지켜야 한다. 면책 고지의 *노출 위치*는 전 핵심 화면 상시 푸터로 확정(FR-037)했으나, 법무 검토를 거친 정식 면책·고지 문안 정책은 아직 미정.
- **장세 판정 모델의 신뢰도:** 9단계 판정과 근거 점수의 산출 로직·검증 방법이 미정. SC-005의 소표본 신뢰도 기준도 함께 정의 필요.
- **알림 발송 인프라:** SMS·이메일 발송 채널, 인증·수신 동의, 발송 빈도(장중 3회) 운영 정책 미정.
- **개인화 학습 범위:** 선택적 파인튜닝의 실제 구현 방식·데이터 보관·삭제 보장(FR-028)은 P2이며, 개인정보·동의 처리 설계가 선행되어야 한다.
- **AI 판단 비용/지연:** 종합 판단을 LLM 기반으로 생성할 경우 응답 지연·비용이 "3분 안 결론(SC-001)"과 장중 실시간성에 영향을 줄 수 있다. 모델·캐싱 전략 검토 필요.
- **관리자 인증/권한 관리:** 최초 관리자 시드, 역할(관리자/일반) 관리, 신청 처리 감사 로그 정책이 미정. 인증 구현 방식(Supabase Auth 등)과 정합 필요.
- **신청 본인 확인·개인정보:** 이용 신청 단계의 본인 인증 수준(연락처 인증 필수 여부)과 신청 정보 수집·보관·삭제 정책이 미정. 기존 알림 연락처 인증(FR-011)과 중복·정합 정리 필요.
- **재신청·만료 정책:** 거절 후 재신청 허용 여부, 재신청 대기 기간, 장기 미접속 권한 만료 등 라이프사이클 정책 미정.
